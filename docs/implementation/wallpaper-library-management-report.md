# 壁纸库管理实现报告

日期：2026-07-20

## 契约

用户可以：

- 在壁纸卡片悬停或键盘聚焦后，通过 36 × 36 快捷入口设置桌面壁纸；点击卡片本身只进入详情。
- 从壁纸库菜单进入批量管理，或在全量重新扫描后清理失效记录。
- 在详情页移除单项，在批量模式移除所选项；三条路径都必须先确认。
- 移除运行中壁纸，并由应用停止使用该壁纸的播放目标。
- 在显示器面板中编辑布局草稿；只有点击“应用”才提交，取消、Esc 或外部点击均丢弃草稿。
- 在软件内预览视频；图片详情只显示图片适用的画面设置。

用户不能：

- 通过上述操作删除、移动或修改源文件。
- 在取消确认后改变选择、播放目标或壁纸库记录。
- 用旧的 `missing` 状态替代清理前的全库路径重扫。
- 在异步保存尚未结束时重复提交同一个导入、播放、分类、设置或移除命令。

## 原因与修复层级

| 字段     | 结论                                                           |
| -------- | -------------------------------------------------------------- |
| 症状     | 壁纸只能导入，无法维护库记录；批量危险按钮视觉过重且拥挤。     |
| 局部原因 | 前端没有库级入口与确认状态，后端仅提供单项移除命令。           |
| 通用原则 | 库记录删除必须明确不触碰源文件，并把强危险样式保留给最终确认。 |
| 影响类别 | 单项移除、批量移除、失效清理。                                 |
| 修复层级 | class repair。                                                 |
| 反例证明 | 分类删除仍只管理分类；图片与视频详情继续共用同一移除契约。     |

## 实现

- `WallCore::refresh_missing` 每次检查全部媒体路径；`remove_many` 先验证全部 ID，再原子移除记录和播放分配。
- Tauri 新增 `scan_library`、`remove_media_batch`、`remove_missing_media`；清理确认后仍会再次重扫。
- 媒体库命令先构建候选状态并锁定播放器，持久化成功后才切换共享状态；失败不会留下前端不可见的内存改动。
- 分类、显示布局、播放、声音和设置同样使用候选状态；播放器、开机启动或存储失败时恢复外部副作用，发布失败只记录日志。
- 详情页暂停与停止使用壁纸级命令，一次更新该壁纸的全部显示目标，避免逐屏请求出现部分成功。
- 托盘和后台监视器使用的全局/单目标暂停也走候选状态提交；播放器或存储失败会恢复已应用目标。
- 播放与显示器刷新回滚会记录变更前实际运行的目标，不会重启错误或原本未运行的目标。
- 壁纸库新增“管理”菜单、批量“移出库”轻量操作和两类确认弹窗；详情页新增单项确认弹窗。
- 命令失败会留在当前确认弹窗并显示错误；导入、移除和清理成功均提供三秒提示。
- 库卡片新增低干扰快捷设置按钮，移除双击播放；批量成功后退出批量模式并清空选择。
- 显示器面板采用草稿后提交，校验在线目标和复制/铺展的最少屏幕数；失败可在原面板重试。
- 视频保留应用内预览，图片隐藏所有视频专属控制；详情布局精确对齐 Figma 的 620/384 双栏。
- 设置、分类、播放和外部操作在请求期间禁用重复提交；应用显示布局时也锁定导入和管理入口。
- 下拉菜单和确认弹窗补齐键盘导航、Esc、初始焦点、焦点陷阱和关闭后焦点返回；弹窗打开时背景不可交互。
- 托盘继续复用统一播放快照更新入口，并使用 Figma 的 Active、Paused、Idle 三态资源。

## 托盘图标可见性修复

| 字段     | 结论                                                              |
| -------- | ----------------------------------------------------------------- |
| 症状     | Windows 通知区中的 Wall 图标比相邻图标小，运行与暂停角标不明显。   |
| 局部原因 | 24 × 24 资源的非透明主体仅为 20 × 20，系统缩放后约剩 13 × 13。     |
| 通用原则 | 小尺寸系统图标应使用一致的安全边距，并以形状和色彩共同表达状态。   |
| 影响类别 | Idle、Active、Paused 三个托盘状态。                                |
| 修复层级 | class repair；统一放大三个资源，不改变播放状态模型或更新入口。     |
| 反例证明 | 部分目标暂停但仍有目标播放时继续显示 Active，而不是 Paused。       |

三个 Figma 原组件保持 24 × 24 和原节点 ID，主体统一放大为 22 × 22；运行/暂停角标使用约 7px 的绿色/黄色
标记与深色分隔圈。Rust 资源测试固定三个 RGBA 的非透明边界，防止后续导出再次缩小。

## 剩余风险

- Windows 通知区的最终相邻图标体量仍需用户人工确认；自动化无法从应用窗口截取系统托盘区域。
