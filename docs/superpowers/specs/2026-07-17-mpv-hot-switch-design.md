# mpv 热切换设计

<!-- 本文记录 Wall 视频壁纸切换延迟修复的已确认设计与验收边界。 -->

## 问题与目标

当前每次切换壁纸都会创建新的 mpv 进程，依次等待窗口创建、WorkerW 嵌入、命名管道就绪、运行时设置和旧进程退出。
本机实测新进程约 180ms 创建，IPC 约 877ms 就绪，首个播放位置约 1.34s 才出现；连续快速切换还会长时间占用
同步 Tauri 命令。

目标是在显示目标和播放器启动参数兼容时复用现有 mpv，通过 IPC 原地加载新媒体，消除重复进程启动和桌面嵌入。

## 方案比较

1. 复用每个显示目标的 mpv，通过 `loadfile <path> replace` 热切换。改动集中、资源稳定，推荐采用。
2. 继续创建新进程，但在首帧后交换窗口。能保留故障隔离，但仍承担进程、窗口和硬件解码初始化延迟。
3. 维护预热进程池。可能进一步降低冷启动，但显著增加资源占用、回收和多屏同步复杂度，v1 不需要。

## 行为设计

- 目标标识、显示模式、显示器集合、屏幕区域、媒体类型和硬件解码设置均未变化时，复用目标现有的全部 mpv 实例。
- 视频切换到视频、图片切换到图片可以热切换；视频与图片互切仍重启，以保持图片无限显示启动参数正确。
- 独立和铺展模式向单个实例发送 `loadfile`；复制模式向组内每个实例发送，并重新归零和同步暂停状态。
- 新媒体加载后通过现有 IPC 路径应用缩放、画幅、抗锯齿、帧率、静音和音量。
- 显示模式、屏幕集合、屏幕区域、媒体类型或硬件解码变化时，沿用现有安全重启路径。
- 热切换成功后更新播放器组的媒体标识、路径、类型和有效设置；进程 PID 与桌面宿主窗口保持不变。
- 用户快速重复操作仍串行进入现有播放器互斥锁，不增加队列、进程池或预加载缓存。

## 错误与恢复

- `loadfile` 使用带请求标识的 IPC 请求并检查 mpv 返回的 `error` 字段，不能只确认管道写入成功。
- 复制组部分实例加载失败时，已切换实例重新加载旧媒体并恢复旧设置；核心快照继续沿用现有回滚逻辑。
- 回滚也失败时返回可恢复播放错误并写入现有日志，不伪报新壁纸已运行。
- mpv 已退出、管道不可用或兼容条件不满足时，不尝试热切换，改走现有重启路径。

## 验收标准

- 兼容的视频切换保持同一个 mpv PID，核心快照和界面显示新壁纸。
- 本机三个测试视频连续切换时，命令提交不再等待新进程和 WorkerW 窗口创建，且无长尾界面未响应。
- 媒体类型、硬件解码或显示布局不兼容时仍创建新进程，原有独立、复制和铺展行为不回退。
- `cargo test`、Clippy、前端测试、生产构建和便携 release 构建通过。
- 使用 Windows UI Automation 完成真实 release 切换，记录 PID 复用和提交耗时，并恢复用户原会话数据。
